# ADR-0005: Session Connect Mechanism for Agent-User Binding

## Status

Accepted

## Date

2026-05-03

## Context

When a runtime agent submits an action request to make-it-so, the control
plane must identify the initiating user and know which mis-client device to
route the resulting approval request to.

Without an explicit initialization step, the agent has no reliable way to
supply this information. The control plane cannot route approvals correctly,
and there is no audit trail linking agent sessions to human users.

The system must solve three related problems:

1. **User identification**: the agent must be able to reference the correct
   user when submitting action requests.
2. **Client routing**: the control plane must know which registered mis-client
   device to use for approval delivery.
3. **Session continuity**: when a user opens a new chat session with an agent,
   the connect overhead should be minimal if a previous binding exists.

The mechanism must work in chat-based interfaces, where the primary input is
text and QR-code scanning is not practical.

---

## Decision Drivers

- low friction in chat contexts (single short phrase, not a URL or token)
- explicit per-session client selection (user chooses which device to use)
- no standing authorization — connect is required before the first action
- session continuity without re-pairing when the binding is still valid
- unknown or stale sessions must be rejected, not silently accepted
- override and reuse events must be logged for audit

---

## Considered Options

1. User supplies a long-lived API key or token directly in chat
2. Out-of-band QR-code scan between mis-client and agent interface
3. Short-lived human-readable pairing code generated by the mis-client

---

## Option Analysis

### Option 1: Long-lived API key or token

#### Advantages
- no pairing step required after initial setup
- works in any interface

#### Disadvantages
- long-lived secrets in chat context are a security risk
- leaking the key grants persistent access to submit requests as that user
- no explicit per-session client selection

#### Assessment
Unacceptable. A long-lived secret in a chat interface violates the
security model and provides no session-scoped audit trail.

---

### Option 2: QR-code scan

#### Advantages
- no manual input required
- strong binding between physical device and session

#### Disadvantages
- not practical in text-based chat interfaces
- requires camera access and a rendered QR-code, which agents cannot display
- breaks the primary chat-based interaction model

#### Assessment
Not viable for the target interaction context.

---

### Option 3: Short-lived human-readable pairing code (selected)

#### Advantages
- easy to type in a chat interface
- code is generated by the authenticated mis-client, anchoring trust to the device
- short TTL and uniqueness window prevent replay and collision
- user explicitly chooses which client generates the code, making the
  per-session client selection intentional
- the code carries no standing authorization; its only purpose is the handshake

#### Disadvantages
- requires the user to switch briefly to their mis-client to generate the code
- relies on uniqueness guarantee within a time window

#### Assessment
Best fit for chat-based agent interactions. Low friction, no persistent
secrets, explicit client selection.

---

## Decision

Adopt the **short-lived human-readable pairing code** as the Session Connect
mechanism.

### Connect Flow

1. User opens their mis-client and triggers "Open Channel".
2. mis-client fetches a short-lived pairing code from the mis-backend.
   The code encodes `(user_id, client_id)` and is valid for ~30 seconds.
3. The mis-backend guarantees uniqueness within a ±30 second window.
4. The code is displayed to the user as a human-readable wordpair
   (e.g., "durstiger-affe"), adapted to the user's UI language.
5. The user types the code into the agent chat.
6. The agent submits `{ code, session_id }` to the mis-backend.
7. The mis-backend validates the code, resolves `(user_id, client_id)`,
   and stores `session_id → (user_id, client_id)`.
8. The agent receives and persistently stores `user_ref` (and `client_id`
   for UX purposes).

### Session Continuation

When a user opens a new chat session and the agent has a stored binding
from a previous session:

1. The agent detects the stored `(user_ref, client_id)` and asks the user
   whether to continue with the existing channel.
2. If confirmed, the agent submits `{ user_ref, client_id, new_session_id }`
   to the mis-backend without requiring a new pairing code.
3. The mis-backend validates that `(user_id, client_id)` is still a
   registered active pair. If the client has been deregistered, the request
   is rejected and the agent recommends a fresh Connect.
4. On successful continuation, the mis-client receives a notification:
   "Neue Session gestartet — Channel open." This is a security signal that
   allows the user to detect unexpected session reuse.

### Data Model

```
mis-backend:
  registered_clients:   user_id  →  [client_id, ...]
  active_sessions:      session_id  →  (user_id, client_id)

agent (persistent):
  user_ref + client_id
```

### Unknown Session Rejection

The mis-backend rejects action requests whose `session_id` has no active
binding. The agent surfaces a message recommending a fresh Connect.

### Override and Logging

A new Connect for an existing agent session overwrites the previous binding.
This is intentional: client selection is explicit and per-session.

All binding changes (new connects, overrides, continuations) are logged
in the mis-backend (pseudonymized) and in the MCP server context where
the session is active.

---

## Consequences

### Positive Consequences

- agents can identify users and route approvals without managing secrets
- client selection is explicit and intentional at session start
- session continuity reduces friction for returning users
- unknown sessions are rejected, preventing unauthorized action requests
- override and reuse events are auditable

### Negative Consequences

- users must switch to their mis-client to generate a pairing code for new
  sessions or explicit client changes
- the mis-backend must guarantee pairing code uniqueness within a time window

### Follow-up Implications

- client registration and deregistration (account-level management of the
  `registered_clients` list) requires a separate, more strongly secured flow
  — this is not defined by this ADR
- policy for maximum session age and forced re-connect may be defined in
  a future ADR
- multiple simultaneous sessions per user are supported; each session
  independently selects a client

---

## Rationale Summary

A short-lived human-readable pairing code generated by the authenticated
mis-client provides the best balance of usability and security for chat-based
agent interfaces. It avoids long-lived secrets, enables explicit per-session
client selection, and anchors trust to the device already used for approvals.
Session continuation keeps friction low for returning users while the
continuation notification preserves auditability.
